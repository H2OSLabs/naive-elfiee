## 总则

### 核心原则

**Lovable 的职责边界**：
- ✅ **负责**：前端 UI 组件、样式、布局、交互动画
- ❌ **不负责**：Tauri 后端、Rust 代码、类型绑定生成、状态持久化

**关键约束**：
1. **类型定义以 `src/bindings.ts` 为准**（由 Tauri Specta 自动生成）
2. **不修改任何 Tauri 调用逻辑**（`invoke` 调用仅在自定义 hooks 中）
3. **不创建新的数据模型类型**（除非纯前端 UI 状态）
4. **所有后端交互通过预定义的 hooks/utils 进行**

### ⚠️ 受限编辑（需遵守规范）

#### 1. 页面主组件 (`src/pages/Index.tsx`)

**可修改部分**：
```typescript
// ✅ UI 结构和样式
return (
  <div className="h-screen flex flex-col bg-background">
    <AppHeader viewMode={viewMode} onViewModeChange={setViewMode} />
    {viewMode === "document" ? (
      <DocumentView {...props} />
    ) : (
      <ConsoleView {...props} />
    )}
  </div>
)
```

**禁止修改部分**：
```typescript
// ❌ 不要修改状态管理和 Tauri 调用
const [blocks, setBlocks] = useState<Block[]>([])

useEffect(() => {
  // ❌ 禁止修改 invoke 调用
  invoke<Block[]>('get_all_blocks', { fileId: currentFileId })
    .then(setBlocks)
}, [currentFileId])
```

**正确做法**：将后端逻辑抽取到自定义 hooks
```typescript
// src/hooks/useElfieeBlocks.ts (手动维护，Lovable 不碰)
export const useElfieeBlocks = (fileId: string) => {
  const [blocks, setBlocks] = useState<Block[]>([])

  useEffect(() => {
    invoke<Block[]>('get_all_blocks', { fileId })
      .then(setBlocks)
  }, [fileId])

  return blocks
}

// src/pages/Index.tsx (Lovable 可编辑)
import { useElfieeBlocks } from '@/hooks/useElfieeBlocks'

const Index = () => {
  const blocks = useElfieeBlocks(currentFileId) // ✅ 只需一行调用

  return <DocumentView blocks={blocks} /> // ✅ 可自由修改 UI
}
```

#### 2. 类型定义 (`src/types/`)

**规则**：
- ✅ 可创建 **纯前端 UI 状态类型**
- ❌ 禁止创建与后端数据模型重复的类型

**示例**：
```typescript
// ✅ 允许 - 纯前端 UI 状态
export type ViewMode = "document" | "console"

export interface UIState {
  sidebarCollapsed: boolean
  drawerOpen: boolean
  selectedBlockId: string | null
}

// ❌ 禁止 - 与 bindings.ts 重复
export interface Block {  // ❌ 已在 bindings.ts 中定义
  id: string
  type: string
  content: string
}
```

**正确做法**：从 `bindings.ts` 重新导出
```typescript
// src/types/index.ts
export { Block, Event, Editor, Capability } from '@/bindings'

// 只添加前端专属类型
export type ViewMode = "document" | "console"
```

## 禁止编辑区域

### ❌ 完全禁止（手动维护）

#### 1. Tauri 后端代码

```
src-tauri/              ← 完全不可编辑
├── src/
│   ├── main.rs
│   ├── lib.rs
│   ├── engine/
│   ├── models/
│   ├── extensions/
│   └── commands/
├── Cargo.toml
└── tauri.conf.json
```

**原因**：Lovable 不支持 Rust，任何修改会破坏后端逻辑。

---

#### 2. 自动生成的类型绑定

```
src/bindings.ts         ← 严禁手动编辑
```

**警告标识**：
```typescript
// This file was generated by [tauri-specta](https://github.com/oscartbeaumont/tauri-specta).
// Do not edit this file manually.
```

**原因**：
- 由 `pnpm tauri dev` 自动生成
- 手动修改会在下次构建时被覆盖
- 类型定义必须与 Rust 后端保持一致

**如需修改类型**：
1. 修改 Rust 源代码 (`src-tauri/src/models/*.rs`)
2. 运行 `pnpm tauri dev` 重新生成
3. **不要直接编辑 `bindings.ts`**

---

#### 3. 自定义 Hooks (`src/hooks/`)

**完全手动维护的 hooks**：
```typescript
// src/hooks/useElfieeBlocks.ts
// src/hooks/useElfieeEvents.ts
// src/hooks/useElfieeCapabilities.ts
// src/hooks/useElfieeCommand.ts
```

**原因**：
- 包含 Tauri `invoke` 调用
- 状态管理逻辑
- 错误处理机制

**Lovable 的角色**：
- ✅ 可以 **使用** 这些 hooks
- ❌ 禁止 **修改** hooks 内部实现

**示例**：
```typescript
// ✅ Lovable 可以这样使用
const DocumentView = () => {
  const blocks = useElfieeBlocks(fileId)  // ✅ 调用 hook

  return <div>{blocks.map(...)}</div>    // ✅ 渲染 UI
}

// ❌ Lovable 不要修改 hook 内部
const useElfieeBlocks = (fileId: string) => {
  // ❌ 这里的逻辑禁止 Lovable 编辑
  useEffect(() => {
    invoke<Block[]>('get_all_blocks', { fileId })
      .then(setBlocks)
  }, [fileId])
}
```

---

#### 4. 工具函数 (`src/utils/`)

**手动维护的工具**：
```typescript
// src/utils/tauriCommands.ts  ← Tauri 命令封装
// src/utils/eventStore.ts     ← Event 处理逻辑
// src/utils/vectorClock.ts    ← 冲突检测
```

**Lovable 可编辑的工具**：
```typescript
// src/utils/formatting.ts     ← 纯前端格式化
// src/utils/validation.ts     ← UI 表单验证
```

**判断标准**：
- 包含 `invoke` 调用 → **禁止编辑**
- 纯前端逻辑（字符串处理、日期格式化） → **可编辑**

## 工作流程

### Git 分支策略

```
main (dev)
├── feat/lovable-ui-redesign  ← Lovable 连接此分支（双向同步）
├── feat/tauri-integration    ← 手动开发 Tauri 绑定
└── feat/agent-extension      ← 后端扩展开发
```

### 开发流程

#### Phase 1: Lovable 生成初始代码

1. **在 Lovable 中**：
   - 导入 GitHub 仓库 `H2OSLabs/elfiee`
   - 选择分支 `feat/lovable-ui-redesign`
   - 提交 `elfiee-lovable-prompt-v5-final.md`
   - 生成初始组件代码

2. **自动同步到 GitHub**：
   ```bash
   # Lovable 自动推送到 feat/lovable-ui-redesign 分支
   ```

3. **本地拉取**：
   ```bash
   git checkout feat/lovable-ui-redesign
   git pull origin feat/lovable-ui-redesign
   ```

---

#### Phase 2: 手动集成 Tauri（开发者）

1. **创建集成分支**：
   ```bash
   git checkout -b feat/tauri-integration
   ```

2. **添加 Tauri 调用**：
   ```typescript
   // src/hooks/useElfieeBlocks.ts (新建)
   import { invoke } from '@tauri-apps/api/core'
   import { Block } from '@/bindings'

   export const useElfieeBlocks = (fileId: string) => {
     const [blocks, setBlocks] = useState<Block[]>([])

     useEffect(() => {
       invoke<Block[]>('get_all_blocks', { fileId })
         .then(setBlocks)
         .catch(console.error)
     }, [fileId])

     return blocks
   }
   ```

3. **更新 Index.tsx**：
   ```typescript
   // src/pages/Index.tsx
   import { useElfieeBlocks } from '@/hooks/useElfieeBlocks'

   const Index = () => {
     // 删除 mock 数据
     // const [blocks, setBlocks] = useState(mockBlocks) ❌

     // 使用 Tauri hook
     const blocks = useElfieeBlocks(currentFileId) // ✅

     // UI 部分保持不变
     return <DocumentView blocks={blocks} />
   }
   ```

4. **生成类型绑定**：
   ```bash
   cd src-tauri
   cargo build  # 生成 src/bindings.ts
   ```

5. **提交并合并**：
   ```bash
   git add .
   git commit -m "feat: integrate Tauri bindings for block management"
   git push origin feat/tauri-integration

   # 创建 PR: feat/tauri-integration → feat/lovable-ui-redesign
   ```

---

#### Phase 3: UI 迭代（Lovable + 开发者）

**场景 1：调整 UI 样式**
- **执行者**：Lovable 或设计师
- **操作**：在 Lovable 中用自然语言 prompt 修改
- **示例 Prompt**：
  ```
  "将 BlockHUD 的背景改为半透明黑色，鼠标悬停时淡入，过渡时间 0.2 秒"
  ```

- **执行者**：Lovable
- **操作**：描述组件需求，Lovable 生成代码
- **示例 Prompt**：
  ```
  "创建一个 BlockConflictIndicator 组件，当 block.hasConflict 为 true 时，

  ```

**场景 3：修改后端逻辑**
- **执行者**：开发者（手动）
- **操作**：
  1. 本地修改 `src-tauri/src/commands/block.rs`
  2. 重新生成 `bindings.ts`
  3. 更新相关 hooks（如 `useElfieeBlocks.ts`）
  4. 提交 PR 到 `feat/lovable-ui-redesign`
  5. Lovable 自动拉取更新

---

#### Phase 4: 合并到主分支

# 4. 解决冲突（如果有）

## 命名和代码规范

### 组件命名

**Lovable 创建的组件必须遵循**：

```typescript
// ✅ 使用 PascalCase + 描述性名称
export const BlockItem = () => {}
export const DocumentSidebar = () => {}
export const MiniTimeline = () => {}

// ❌ 避免
export const Block = () => {}        // 与类型名冲突
export const block_item = () => {}   // 不是 PascalCase
export const BI = () => {}           // 缩写不清晰
```

### 文件命名

```
src/components/
├── BlockItem.tsx          ✅ 组件文件 PascalCase
├── DocumentSidebar.tsx    ✅
├── ui/
│   └── button.tsx         ✅ shadcn 组件小写

src/hooks/
├── useElfieeBlocks.ts     ✅ hooks 使用 camelCase
├── useElfieeCommand.ts    ✅

src/utils/
├── formatting.ts          ✅ 工具文件 camelCase
```

### Props 命名

```typescript
// ✅ 使用 on + 动作名
interface BlockItemProps {
  onEdit: (blockId: string, content: string) => void
  onDelete: (blockId: string) => void
  onRun: (blockId: string) => void
}

// ❌ 避免
interface BlockItemProps {
  editHandler: () => void      // 不清晰
  handleDelete: () => void     // 冗余
  click: () => void            // 太泛化
}
```

### 样式类名

```typescript
// ✅ 使用 TailwindCSS 实用类
<div className="flex items-center gap-2 p-4 bg-card rounded-lg">

// ✅ 复杂样式抽取到 CSS 模块
<div className="block-item-container">

// ❌ 避免内联样式（除非动态计算）
<div style={{ padding: '16px' }}>  // 应使用 className="p-4"
```

## 冲突预防机制

### 1. 类型冲突

**问题**：Lovable 生成的类型与 `bindings.ts` 冲突

**预防**：
```typescript
// ❌ Lovable 不要创建这些类型
export interface Block { ... }
export interface Event { ... }
export interface Editor { ... }

// ✅ 创建别名或扩展类型
import { Block } from '@/bindings'

export type BlockWithUIState = Block & {
  isSelected: boolean      // 前端专属状态
  isHovered: boolean
}
```

**检查命令**：
```bash
# 运行前端构建，检查类型冲突
pnpm run build

# 预期输出：无 TypeScript 错误
```

---

### 2. 文件覆盖

**问题**：Lovable 重新生成组件时覆盖手动添加的 Tauri 调用

**预防策略**：

#### 策略 A：分离逻辑和 UI（推荐）
```typescript
// src/hooks/useBlockEditor.ts (手动维护)
export const useBlockEditor = (blockId: string) => {
  const handleEdit = (content: string) => {
    invoke('edit_block', { blockId, content })  // Tauri 调用
  }
  return { handleEdit }
}

// src/components/BlockItem.tsx (Lovable 可编辑)
import { useBlockEditor } from '@/hooks/useBlockEditor'

const BlockItem = ({ block }) => {
  const { handleEdit } = useBlockEditor(block.id)  // 只需一行

  return (
    <Card>
      {/* Lovable 可以随意修改 UI */}
      <Button onClick={() => handleEdit(newContent)}>保存</Button>
    </Card>
  )
}
```

#### 策略 B：使用代码注释标记
```typescript
// src/pages/Index.tsx

// === LOVABLE EDITABLE START ===
return (
  <div className="h-screen flex">
    <DocumentView blocks={blocks} />
  </div>
)
// === LOVABLE EDITABLE END ===

// === MANUAL CODE - DO NOT EDIT IN LOVABLE ===
useEffect(() => {
  invoke('get_all_blocks', { fileId })
    .then(setBlocks)
}, [fileId])
// === END MANUAL CODE ===
```

---

### 3. 依赖冲突

**问题**：Lovable 添加的 npm 包与 Tauri 不兼容

**预防**：
```json
// package.json - 锁定关键依赖版本
{
  "dependencies": {
    "@tauri-apps/api": "^2.0.0",      // 锁定主版本
    "react": "^18.3.1",                // 已锁定
    "react-dom": "^18.3.1"
  }
}
```

**Lovable 添加依赖前检查**：
```bash
# 在 Lovable prompt 中说明
"添加新的 UI 库前，请确保与 @tauri-apps/api 2.0 兼容"
```

---

### 4. Git 冲突

**场景**：Lovable 推送的代码与手动开发分支冲突

**解决流程**：
```bash
# 1. 拉取 Lovable 分支
git checkout feat/lovable-ui-redesign
git pull origin feat/lovable-ui-redesign

# 2. 合并手动开发分支
git merge feat/tauri-integration

# 3. 解决冲突 - 优先保留规则
# - bindings.ts: 保留自动生成版本（git checkout --theirs src/bindings.ts）
# - hooks/: 保留手动版本（git checkout --ours src/hooks/）
# - components/: 手动合并（保留 Lovable 的 UI + 手动的逻辑）

# 4. 提交合并
git add .
git commit -m "merge: integrate Tauri hooks with Lovable UI"
git push origin feat/lovable-ui-redesign
```

- 所有 Tauri 调用逻辑
- 自定义 hooks
- 错误处理
- 性能优化（React.memo、useMemo）

## 常见问题 (FAQ)

### Q1: Lovable 生成的代码使用了自定义类型，如何处理？

**错误示例**：
```typescript
// Lovable 生成的 src/types/block.ts
export interface Block {
  id: string
  type: string
  content: string
}
```

**修复步骤**：
1. 删除 Lovable 生成的类型文件
   ```bash
   rm src/types/block.ts
   ```

2. 在 `src/types/index.ts` 中重新导出
   ```typescript
   export { Block } from '@/bindings'
   ```

3. 在 Lovable 中添加说明
   ```
   Prompt: "使用 @/bindings 中的 Block 类型，不要创建新的 Block 接口"
   ```

---

### Q2: 如何在 Lovable 中使用自定义 hooks？

**正确做法**：
```typescript
// 在 Lovable prompt 中说明
"使用 useElfieeBlocks hook 获取 blocks 数据，
 该 hook 已在 src/hooks/useElfieeBlocks.ts 中定义，
 接受 fileId 参数并返回 Block[] 数组"
```

**Lovable 生成的代码**：
```typescript
import { useElfieeBlocks } from '@/hooks/useElfieeBlocks'

const DocumentView = ({ fileId }) => {
  const blocks = useElfieeBlocks(fileId)

  return <div>{blocks.map(...)}</div>
}
```

---

### Q3: Lovable 覆盖了手动添加的代码怎么办？

**预防**：
- 使用 Git 分支隔离（`feat/lovable-ui-redesign` vs `feat/tauri-integration`）
- 将 Tauri 逻辑放在 `src/hooks/` 中（Lovable 禁止修改）

**恢复**：
```bash
# 1. 查看 Lovable 的修改
git diff feat/lovable-ui-redesign

# 2. 如果覆盖了关键代码，回退单个文件
git checkout HEAD~1 -- src/hooks/useElfieeBlocks.ts

# 3. 提交修复
git add .
git commit -m "fix: restore manually maintained hooks"
```

### Q4: 如何让设计师安全地使用 Lovable？

**给设计师的简化指南**：

1. **只修改样式**
   - 颜色、间距、圆角、阴影
   - 动画效果（过渡、淡入淡出）

2. **使用自然语言 Prompt**
   ```
   "将卡片的阴影从 shadow-sm 改为 shadow-lg"
   "鼠标悬停时，按钮背景色变为蓝色，过渡 0.2 秒"
   ```

3. **不要修改**
   - 任何包含 `invoke` 的代码
   - `src/hooks/` 目录
   - `src/bindings.ts` 文件

4. **遇到问题时**
   - 在 Lovable 中点击 "Undo" 撤销
   - 通知开发者 review

## 总结

**核心规则**：
1. **Lovable 负责 UI，开发者负责逻辑**
2. **类型定义以 `bindings.ts` 为准**
3. **所有 Tauri 调用放在 `hooks/` 中**
4. **使用 Git 分支隔离 Lovable 和手动修改**

**成功标志**：
- ✅ `pnpm run build` 无错误
- ✅ `pnpm tauri dev` 正常启动
